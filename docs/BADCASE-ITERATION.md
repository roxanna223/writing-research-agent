# Badcase 迭代记录

> 项目：同人/原创写作资料辅助Agent
> 记录周期：第1轮至第5轮
> 核心优化目标：JSON解析成功率、字段匹配率、输出一致性

---

## 概述

在6步结构化Agent工作流（任务澄清 -> 研究规划 -> 资料检索 -> 设定提取 -> 设定整理 -> 设定交付）中，设定提取（Extractor）和设定整理（Checker/Formatter）环节是决定输出质量的关键路径。这两个环节严重依赖LLM的结构化输出能力——需要从研究资料中提取结构化设定卡，并按统一Schema组装为设定包。

然而，LLM的结构化输出在实践中暴露出一系列健壮性问题。从第1轮到第5轮，我们进行了系统性迭代，将JSON解析成功率从40%提升至90%，字段匹配率从30%提升至85%，并最终通过架构改造将生成稳定率从50%提升至100%。

以下为每一轮的详细记录。

---

## 第1轮：初始版本问题发现

### 版本信息

- 版本号：v0.1.0-alpha
- 核心流程：Storm/Clear模式 -> 研究资料检索 -> 单次LLM调用生成完整设定包JSON
- Prompt策略：自然语言描述输出格式，无强制Schema约束

### 发现的核心问题

#### 问题1：JSON截断

**现象描述**：

当LLM一次性生成包含10+张设定卡的完整JSON时，输出在约第6-7张卡片处被截断。截断点通常位于某个字符串值中间，或数组元素之间，导致整个JSON无法被`JSON.parse()`解析。

```
// 截断示例
{
  "characters": [
    {"name": "林若溪", "age": 24, "role": "主角", "personality": "外冷内热，擅长剑术", ...},
    {"name": "苏远舟", "age": 26, "role": "男主", "background": "出身江湖世家",
    {"name": "顾清寒", "age": 19, "role": "配角", "traits": ["聪慧", "   // <-- 在此截断
```

**根因分析**：

- LLM输出存在token长度限制（当时使用的模型上下文窗口为8K，输出限制约4K token）
- 单次生成完整设定包的数据量超出模型稳定输出范围
- 未在Prompt中对输出长度做任何约束或分段策略
- 没有输出后校验与重试机制

**修复方案**：

- 临时方案：手动补全截断的JSON（仅用于验证流程可行性）
- 计划方案：拆分输出、增加长度约束、增加重试逻辑

#### 问题2：字段漂移

**现象描述**：

同一张设定卡在不同次调用中，表达相同语义的字段使用了不同的字段名。例如：

| 第1次调用 | 第2次调用 | 第3次调用 |
|-----------|-----------|-----------|
| `character_name` | `name` | `charName` |
| `personality_desc` | `personality` | `character_traits` |
| `background_story` | `background` | `backstory` |
| `ability_list` | `skills` | `abilities` |

这导致后续的Formatter和Checker无法可靠地读取字段，组装设定包时大量字段丢失或错位。

**根因分析**：

- Prompt中仅以自然语言描述"需要哪些字段"，未给出精确的字段名规范
- LLM在不同上下文中有不同的命名偏好，对"同义字段"的理解存在随机性
- 缺少Schema校验层，无法在输出后检测和修正字段名

**修复方案**：

- 计划引入严格的JSON Schema定义，强制字段名
- 在Prompt中使用few-shot示例统一命名

#### 问题3：双模式输出无区分

**现象描述**：

Storm模式（创意激荡，面向同人创作）和Clear模式（严谨考据，面向原创设定）在输出格式上完全相同。两种模式虽然Prompt中有不同的"风格描述"，但最终产出的设定卡在结构、字段、内容风格上没有体现出应有的差异。

- Storm模式的预期：开放式的联想、多可能性罗列、鼓励"如果...会怎样"的推理
- Storm模式的实际：与Clear模式一样的确定语气、单答案输出
- Clear模式的预期：基于源材料的精确提取、引用出处、确定结论
- Clear模式的实际：同样没有引用标记，没有区分"canon确认"与"推测"

**根因分析**：

- 两种模式共用同一套输出模板，仅在系统Prompt中做了文字层面的风格区分
- 输出结构没有为两种模式设计差异化字段（如Storm的`alternative_interpretations`、Clear的`source_reference`）
- 没有利用7个Skill的能力差异来体现模式差异

**修复方案**：

- 计划为两种模式设计不同的输出Schema
- Storm模式增加创意字段，Clear模式增加引用字段

### 本轮数据

| 指标 | 数值 |
|------|------|
| JSON解析成功率 | 40% |
| 字段匹配率 | 30% |
| 生成稳定率 | 50%（两次同样输入产出可用的概率） |
| 单次生成平均耗时 | 45秒 |

### 遗留问题

1. 截断问题需要结构性方案，不能依赖临时补全
2. 字段命名需要统一标准并强制约束
3. 双模式差异需要从Schema层面设计
4. 缺少任何形式的输出质量校验

## 第2轮：输出格式约束强化

### 版本信息

- 版本号：v0.2.0-beta
- 核心改动：引入JSON Schema约束 + 输出长度限制

### 修复内容

#### Prompt修改

本轮的策略是"收紧约束"，通过技术手段强制LLM输出符合规范的JSON：

1. **JSON Schema引入**：在Prompt中嵌入完整的JSON Schema定义，明确每个字段的名称、类型、必填/可选、枚举值范围。

```
// 嵌入Prompt的Schema示例片段
{
  "type": "object",
  "required": ["character_id", "name", "role", "personality", "background"],
  "properties": {
    "character_id": {"type": "string", "pattern": "^CHAR-[A-Z0-9]{6}$"},
    "name": {"type": "string", "minLength": 1, "maxLength": 50},
    "role": {"type": "string", "enum": ["主角", "配角", "反派", "路人", "其他"]},
    ...
  }
}
```

2. **输出长度限制**：明确要求"每张设定卡不超过800字"，"单次输出不超过5张设定卡"。

3. **结构化标记**：要求以`` ```json ``标记包裹输出，便于后续解析。

#### 效果评估

**改善方面**：

- JSON截断率明显下降：从60%截断降至约35%截断
- 字段漂移有所缓解：同一Schema约束下，LLM倾向于使用规定的字段名
- 输出格式更统一：所有输出都有了相同的外层结构

**新问题：内容质量下降**

约束过强带来了副作用：

1. **模板化严重**：每张卡读起来像是填表，失去了自然语言的表达力。例如：
   - 约束前：`"林若溪自幼在雪山习剑，十六岁下山时已能一剑破三尺冰。"`
   - 约束后：`"背景：雪山上长大。技能：剑术。"`

2. **创意被压抑**：Storm模式在严格Schema下完全无法发挥创意，`alternative_interpretations`等扩展字段因为Schema中未定义而被LLM主动避免。

3. **字段填充率两极分化**：必填字段被机械填充（有时是"无"或"未知"），可选字段几乎从不出现。

4. **Schema定义本身成为新的失败点**：当Schema过于复杂时，LLM会在嵌套结构中出现字段遗漏。

### 本轮数据

| 指标 | 第1轮 | 第2轮 | 变化 |
|------|-------|-------|------|
| JSON解析成功率 | 40% | 55% | +15% |
| 字段匹配率 | 30% | 50% | +20% |
| 生成稳定率 | 50% | 55% | +5% |
| 内容质量评分（5分制） | 3.5 | 2.8 | -0.7 |

### 遗留问题

1. 约束与质量之间存在明显的权衡（trade-off），需要找到平衡点
2. 单次长JSON输出的截断问题仍未从根本上解决
3. 需要更智能的生成策略，而非简单收紧约束
4. 缺乏"生成后校验"与"修复"环节

## 第3轮：分步生成策略

### 版本信息

- 版本号：v0.3.0-beta
- 核心改动：将长JSON拆分为多次短JSON调用，每张设定卡独立生成

### 修复内容

#### 架构改动

本轮的思路发生了根本性转变：**不再试图让LLM一次生成所有内容，而是将大任务拆解为多个小任务**。

```
[改动前]
研究资料 -> 1次LLM调用 -> 完整设定包JSON（10张卡，~8KB）

[改动后]
研究资料 -> 分维度拆分 -> N次LLM调用（每次1-2张卡，~1KB/次）
         ├── 调用1：生成人物卡1-2
         ├── 调用2：生成人物卡3-4
         ├── 调用3：生成世界观卡1-2
         ├── 调用4：生成时间线卡
         ├── ...
         └── 调用N：最后组装验证
```

#### Prompt修改

1. **单卡生成模式**：每次Prompt仅要求LLM生成1-2张设定卡，大幅降低单次输出的复杂度。

2. **上下文携带**：每次调用时携带先前已生成的卡片摘要（而非完整JSON），让LLM了解已有内容以维持一致性。

3. **独立Schema定义**：为人物卡、世界观卡、时间线卡等不同类型分别定义Schema，每个Schema更精简。

#### 效果评估

**改善方面**：

- JSON截断大幅减少：从35%截断率降至约10%
- JSON解析成功率显著提升：从55%提升至70%
- 单张卡片的质量回升：内容更有深度，不再像填表

**新问题：卡片间一致性变差**

分步生成虽然解决了截断问题，但引入了新的挑战：

1. **人物关系不一致**：
   - 卡A说"林若溪与苏远舟是青梅竹马"
   - 卡B说"林若溪与苏远舟三年前初识"

2. **设定冲突**：
   - 世界观卡写道"这个世界没有魔法"
   - 人物卡中某角色拥有"火系魔法天赋"

3. **时间线断裂**：
   - 事件A发生在"天启三年三月"
   - 事件B发生在"天启三年五月"
   - 但与事件B关联的事件C被写为"天启三年四月"（应在五月之后）

4. **风格不统一**：
   - 不同卡片的写作风格、详细程度参差不齐
   - 同一设定在不同卡片中的描述详略不同

### 本轮数据

| 指标 | 第2轮 | 第3轮 | 变化 |
|------|-------|-------|------|
| JSON解析成功率 | 55% | 70% | +15% |
| 字段匹配率 | 50% | 60% | +10% |
| 生成稳定率 | 55% | 60% | +5% |
| 一致性评分（5分制） | 3.0 | 2.2 | -0.8 |
| 内容质量评分（5分制） | 2.8 | 3.5 | +0.7 |

### 遗留问题

1. 卡片间的一致性需要专门的校验和修复环节
2. 分步生成的上下文传递策略需要优化
3. 需要一个"总装与校验"步骤来发现和修复冲突
4. 这直接催生了第4轮的Checker Skill

## 第4轮：Checker校验 + 自修复

### 版本信息

- 版本号：v0.4.0-rc
- 核心改动：新增Checker Skill，对每张卡进行Schema校验，Extractor根据校验报告修复

### 修复内容

#### 架构改动

在第3轮的分步生成流程之后，新增一个"校验-修复"循环：

```
第3轮流程：
  Extractor（分步生成） -> Formatter（组装）

第4轮流程：
  Extractor（分步生成） -> Checker（Schema校验） -> Extractor（修复） -> Formatter（组装）
                              ↑                                  |
                              └──────── 校验不通过时触发 ─────────┘
```

#### Checker Skill设计

Checker的核心职责是"不看内容只看结构"（在此阶段），确保每一张设定卡在Schema层面是合法的：

1. **必填字段检查**：所有`required`字段是否都存在且非空
2. **类型检查**：字段值的类型是否与Schema定义一致（string/number/array/object）
3. **枚举值检查**：限定了`enum`的字段值是否在允许范围内
4. **格式检查**：`character_id`是否匹配`CHAR-[A-Z0-9]{6}`等格式约束
5. **引用完整性检查**：card之间的引用ID是否能找到对应目标

Checker输出的校验报告格式：

```json
{
  "card_id": "CHAR-A1B2C3",
  "status": "FAIL",
  "issues": [
    {
      "field": "birth_year",
      "severity": "ERROR",
      "message": "birth_year应为number类型，实际为string",
      "expected": "number",
      "actual": "\"不详\""
    },
    {
      "field": "faction",
      "severity": "WARNING",
      "message": "faction值 '散修' 不在枚举列表中",
      "expected": "['正道', '魔教', '中立', '无']",
      "actual": "\"散修\""
    }
  ],
  "fix_suggestions": [
    "将 birth_year 改为 null 或移除该字段",
    "将 faction 改为 '中立' 或在枚举列表中增加 '散修'"
  ]
}
```

#### Prompt修改

1. **Extractor修复Prompt**：接收Checker的校验报告，针对性地修复问题字段。修复时保留原有内容，仅修正结构和命名。

2. **Checker的Prompt**：严格按Schema进行机械校验，不做内容质量判断（那是后续Checker内容校验版本的事）。

#### 效果评估

**改善方面**：

- 字段匹配率大幅提升：从60%提升至75%
- Schema合规率接近100%：只要有校验报告，Extractor基本能修复

**新问题：修复循环无限进行**

Checker发现错误 -> Extractor修复 -> 再次校验 -> 又发现新错误（或因修复引入新错误） -> 再次修复...

这在以下场景中尤为突出：

1. **枚举值不完整**：枚举列表`["剑", "刀", "枪", "弓"]`中不包含"暗器"，修复为"剑"后丢失了原始信息
2. **必填字段的值无法确定**：`birth_year`必须为number，但源材料中确实没有年份信息，修复后填入`0`或`-1`，引入新的语义问题
3. **修复引发连锁反应**：修改一个字段的枚举值触发了对其他相关字段的重新评估

### 本轮数据

| 指标 | 第3轮 | 第4轮 | 变化 |
|------|-------|-------|------|
| JSON解析成功率 | 70% | 78% | +8% |
| 字段匹配率 | 60% | 75% | +15% |
| 生成稳定率 | 60% | 70% | +10% |
| 平均修复轮次 | - | 2.8次 | - |
| 存在无限循环的比例 | - | ~15% | - |

### 遗留问题

1. 修复循环需要上限控制
2. 枚举值固定的策略过于僵化，需要灵活的模糊匹配
3. 部分字段确实无法满足严格的Schema约束时，需要容错策略
4. JSON解析失败时（即使概率已降低）仍缺少降级方案

## 第5轮：容错逻辑 + 兜底策略

### 版本信息

- 版本号：v1.0.0
- 核心改动：容错解析、模糊匹配、重试上限、架构改造（确定性逻辑）
- 这是实现从"能用"到"好用"的关键一轮

### 修复内容

#### 改动1：分层容错解析

JSON解析不再只是一个`JSON.parse()`调用，而是设计了一个三层降级策略：

```
Layer 1: 标准JSON.parse()
    ├── 成功 -> 进入Schema校验
    └── 失败 -> 进入Layer 2

Layer 2: JSON修复 + 重试
    ├── 检测常见错误（尾部逗号、未闭合引号、单引号等）并自动修复
    ├── JSON.parse() 重试
    ├── 成功 -> 进入Schema校验
    └── 失败 -> 进入Layer 3

Layer 3: 正则提取 + 默认值填充
    ├── 用正则表达式从原始文本中提取关键字段值
    ├── 字段名通过模糊匹配（编辑距离）映射到标准Schema字段
    ├── 缺失的必填字段用类型对应的默认值填充
    └── 标记该卡片为 "auto_recovered": true
```

**Layer 3详细实现**：

```python
# 正则提取示例
patterns = {
    "name": r'(?:名字|姓名|名称|name)[：:]\s*["\']?([^"\'\n，,]+)',
    "age": r'(?:年龄|age)[：:]\s*(\d+)',
    "role": r'(?:角色|身份|定位|role)[：:]\s*["\']?([^"\'\n，,]+)',
    "personality": r'(?:性格|personality)[：:]\s*["\']?([^"\'\n]+)',
}

# 编辑距离模糊匹配
def fuzzy_match_field(extracted_key: str, schema_fields: list[str]) -> str:
    """使用Levenshtein距离找到最匹配的schema字段"""
    best_match = min(schema_fields, key=lambda f: levenshtein(extracted_key.lower(), f.lower()))
    distance = levenshtein(extracted_key.lower(), best_match.lower())
    if distance <= 3:  # 阈值可配置
        return best_match
    return None  # 无法匹配，使用默认值

# 默认值策略
DEFAULT_VALUES = {
    "string": "待补充",
    "number": -1,
    "array": [],
    "boolean": False,
}
```

#### 改动2：模糊匹配 + 映射表

在第1轮的"字段漂移"问题上，从"强制约束"转向"灵活包容"：

1. **映射表维护**：建立一个持续更新的字段别名映射表

```
| 标准字段        | 已知别名                                      |
|----------------|----------------------------------------------|
| character_name | name, charName, char_name, 角色名, 姓名        |
| personality    | character_traits, traits, personality_desc, 性格 |
| background     | backstory, background_story, bg, 背景, 身世    |
| abilities      | skills, ability_list, powers, 能力, 技能        |
| appearance     | looks, appearance_desc, 外貌, 形象             |
```

2. **编辑距离自动匹配**：对于映射表中未覆盖的新别名，使用Levenshtein编辑距离（阈值≤3）自动匹配到最近的标准字段。

3. **未知字段保留**：无法匹配的字段不会被丢弃，而是被放入`extra_fields`对象中保留，供人工审核时参考。

#### 改动3：最大重试次数限制

在Checker-Extractor修复循环中增加了硬性上限：

```
修复循环参数：
- max_retries: 3（最大重试次数）
- severity_threshold: ERROR（WARNING级别不触发重试）
- 达到上限后：保留最后一个修复版本，标记为 "repair_exhausted": true
- 所有标记为 repair_exhausted 的卡片在最终交付时附带警告说明
```

这一改动彻底消除了"修复循环无限进行"的问题（第4轮遗留的~15%无限循环）。

#### 改动4：架构改造——放弃端到端生成

这是本轮最根本的改动，也是将"生成稳定率从50%提升至100%"的关键：

**改造前的架构（v0.1 - v0.4）**：

```
研究资料 -> LLM自由生成设定卡JSON -> 校验 -> 修复 -> 组装
              ↑
         核心问题：LLM需要"凭空创造"设定卡的结构和内容
         每一次调用都是不确定的
```

**改造后的架构（v1.0）**：

```
研究资料 -> LLM提取关键信息点（非结构化文本片段）
         -> 确定性逻辑：将信息点按预定义模板填充到设定卡槽位
         -> 设定卡组装（模板化）
         -> Checker校验
         -> Formatter格式化输出
              ↑
         核心变化：LLM只做"理解与提取"（它擅长的）
                  确定性的代码逻辑做"结构化和组装"（它不擅长的）
```

**为什么这个改动让稳定率达到100%**：

- LLM的输出从"完整JSON"变为"结构化的信息点列表"，JSON部分大幅简化
- 卡片组装逻辑是确定性的代码（if-else / template rendering），同一输入必定同一输出
- 字段名、嵌套结构、格式完全由代码控制，不再依赖LLM的随机性
- 当LLM的提取结果不完整时，模板的默认值机制保证卡片结构始终完整

#### Prompt修改

1. **Few-shot示例**：在Extractor Prompt中增加3-5个完整的"研究文本 -> 信息点 -> 设定卡"转换示例，覆盖正常、边界和异常三种场景。

2. **明确边界行为**：
   - 信息不足时的行为：标记`confidence: "low"`而非编造
   - 冲突信息的行为：保留双方信息，标记`conflict: true`
   - 无法分类信息的行为：放入`misc_notes`而非强行归类

3. **模板化输出指令**：不再要求LLM输出完整JSON，而是输出更简单的结构化文本，由代码层完成最终组装。

## 5轮迭代总结

### 数据演进总览

| 指标 | 第1轮 | 第2轮 | 第3轮 | 第4轮 | 第5轮 |
|------|-------|-------|-------|-------|-------|
| JSON解析成功率 | 40% | 55% | 70% | 78% | **90%** |
| 字段匹配率 | 30% | 50% | 60% | 75% | **85%** |
| 生成稳定率 | 50% | 55% | 60% | 70% | **100%** |

### 关键经验教训

1. **约束 vs 灵活的平衡**：第2轮的"强约束"策略在第3-5轮被逐步修正为"容错+模糊匹配"策略。结论：与LLM协作时，硬约束往往不如弹性容错有效。

2. **分而治之优于一次成型**：第3轮的"分步生成"证明了将复杂任务拆解为简单子任务是应对LLM输出不确定性的有效策略。

3. **校验闭环是质量保障的关键**：第4轮的Checker-Extractor循环构成了一个基础的质量保障闭环，但需要上限控制来防止失控。

4. **LLM做擅长的事，代码做确定的事**：第5轮的架构改造是最根本的突破。LLM擅长理解自然语言和提取信息，但不擅长保证结构化的精确性。将结构化工作交给确定性代码逻辑后，稳定性从根本上得到解决。

5. **容错设计是生产级系统的标配**：第5轮的三层容错解析策略使得系统在最坏情况下也能产出"部分可用"的结果，这是从demo到产品的关键跨越。
