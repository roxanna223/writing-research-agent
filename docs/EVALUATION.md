# 6维度评估体系

> 项目：同人/原创写作资料辅助Agent
> 版本：v1.0
> 评估目标：全面衡量设定包的输出质量

---

## 评估体系概述

本评估体系涵盖6个核心维度，覆盖从内容质量到工程健壮性的完整质量视图。评估采用A/B对照设计（本Agent vs ChatGPT），双通道评审（人工 + LLM评审），多题多评委交叉验证。

### 维度权重总览

| 维度 | 权重 | 核心问题 |
|------|------|----------|
| 研究维度覆盖率 (Coverage) | 25% | 设定包覆盖了哪些维度？有没有遗漏？ |
| 设定准确性 (Accuracy) | 20% | 与源材料/canon是否一致？有没有编造？ |
| 设定一致性 (Consistency) | 20% | 设定包内部有没有逻辑矛盾？ |
| 创意丰富度 (Creativity) | 15% | 在canon基础上是否有合理的延伸？ |
| 格式规范性 (Format) | 10% | 输出是否结构化、可读？ |
| 可用性/可复用性 (Usability) | 10% | 设定包能否直接用于后续写作？ |

### 综合得分公式

```
综合得分 = Coverage×0.25 + Accuracy×0.20 + Consistency×0.20 + Creativity×0.15 + Format×0.10 + Usability×0.10
```

得分范围：1.00 - 5.00

**当前系统综合得分：4.5/5**（与ChatGPT做同题A/B对照结果）

## 维度1：研究维度覆盖率 (Coverage)

**权重：25%**

### 定义

衡量设定包对写作所需研究维度的覆盖完整程度。维度包括但不限于：

| 类别 | 具体维度 |
|------|----------|
| 人物 | 主要角色、次要角色、人物关系网络、角色弧线 |
| 世界观 | 世界规则、社会结构、政治体系、经济系统 |
| 时间线 | 关键事件序列、时代背景、历史脉络 |
| 地理 | 地图信息、区域特征、地点关联 |
| 文化 | 语言、习俗、宗教、艺术、价值观 |
| 科技/魔法 | 技术/魔法体系、能力等级、使用规则 |
| 组织 | 势力分布、组织架构、派系关系 |
| 物品 | 关键道具、装备、文物 |

### 评分 Rubric

| 分数 | 标准 | 典型描述 |
|------|------|----------|
| **5** | 全覆盖 | 8个类别维度中至少7个有实质性内容；每个有内容的维度包含3个以上具体条目；人物关系网络完整（含动态关系描述） |
| **4** | 核心+扩展 | 8个维度中6个有实质性内容；关键维度（人物、世界观、时间线）内容充实；有2个以上维度包含延伸信息 |
| **3** | 核心覆盖 | 8个维度中5个有内容；人物维度至少包含主要角色；世界观维度有基本框架；但部分维度仅有1-2个条目，深广度不足 |
| **2** | 部分覆盖 | 8个维度中3-4个有内容；存在明显的维度盲区（如缺少时间线或文化）；已覆盖维度的细节不足 |
| **1** | 严重不足 | 8个维度中2个及以下有内容；仅覆盖最基础的人物信息；无法支撑后续写作 |

### 数据采集方式

- 自动统计：解析设定包，统计各维度的条目数量和字段填充率
- 人工评估：对照写作需求，判断缺失的维度是否"关键遗漏"

### 项目数据

- **研究维度覆盖率：100%**（8个维度全量覆盖，每个维度有3+具体条目）

## 维度2：设定准确性 (Accuracy)

**权重：20%**

### 定义

衡量设定包中的信息与源材料（原作/canon/用户提供的参考资料）的一致程度。核心判断标准：

- 是否忠实反映了源材料中的已知信息？
- 是否存在与canon直接冲突的内容？
- 是否存在无源材料支撑的"幻觉"编造？
- 推测性内容是否被正确标注（vs 伪装成已知事实）？

### 评分 Rubric

| 分数 | 标准 | 典型描述 |
|------|------|----------|
| **5** | 高度准确 | 所有canon信息准确无误；推测性内容均有明确标注（如`confidence: low` / `source: inferred`）；可以逐条追溯信息来源 |
| **4** | 基本准确，少量瑕疵 | 核心canon信息准确；存在1-2处不影响主线的细节偏差；大部分推测有标注但偶有遗漏 |
| **3** | 有可接受偏差 | canon主体正确；存在3-5处细节错误或未标注的推测；不影响整体可用性但需要写作者自行核查 |
| **2** | 存在显著错误 | canon信息出现重要错误（如角色关系颠倒、事件时序错误）；大量推测未标注；需要人工大幅修正后才能使用 |
| **1** | 严重失实 | canon信息大面积错误；虚构内容伪装为canon事实；基本不可用，需重新检索 |

### 检查项清单

- [ ] 角色名称、关系是否与源材料一致
- [ ] 时间线事件顺序是否正确
- [ ] 世界规则是否与canon设定一致
- [ ] 推测性内容是否有`confidence`标记
- [ ] 推测性内容是否有`source`或`basis`说明
- [ ] 数值型信息（年龄、年份、距离等）是否准确
- [ ] 引用出处是否存在"张冠李戴"

### 数据采集方式

- 事实核查：随机抽取20%的设定条目，逐条比对源材料
- 幻觉得分：统计未标注推测但无法从源材料验证的条目比例
- 标注完整率：推测性条目中有标注的比例

## 维度3：设定一致性 (Consistency)

**权重：20%**

### 定义

衡量设定包内部是否存在逻辑矛盾。一致性问题的类型包括：

| 冲突类型 | 示例 |
|----------|------|
| 角色关系冲突 | 卡A说A与B是兄妹，卡B说A与B是青梅竹马（非血缘） |
| 时间线冲突 | 事件X发生在"三年前"，事件Y发生在"五年前"，但X被描述为Y的"后续事件" |
| 规则冲突 | 世界观卡说"魔法已失传"，人物卡中某角色拥有"火系魔法天赋" |
| 数值冲突 | 人物卡写年龄25岁，时间线推算应为32岁 |
| 等级/能力冲突 | 设定为"初学者"的角色拥有"大师级"技能 |
| 地理位置冲突 | 两个地点A和B被描述为"相邻"，但分别在两块大陆 |
| 权力/组织冲突 | 同一个人物在不同卡中隶属于不同组织 |

### 评分 Rubric

| 分数 | 标准 | 典型描述 |
|------|------|----------|
| **5** | 完全一致 | 所有设定卡之间的交叉信息无任何逻辑矛盾；跨卡引用准确无误；时间线自洽；规则体系内无冲突 |
| **4** | 基本一致，微小瑕疵 | 存在1-2处低影响冲突（如非关键配角的次要信息不一致）；核心设定（主线人物、主要世界观规则）完全自洽 |
| **3** | 存在若干冲突但不影响主线 | 存在3-5处冲突，其中至少1处涉及主要设定；需要写作者在创作时做出取舍和修正 |
| **2** | 冲突影响可用性 | 存在5处以上冲突，或存在关键冲突（如主角设定矛盾、核心世界观规则相互否定）；需系统性修正 |
| **1** | 严重矛盾 | 大面积自相矛盾；无法判断哪些信息是"正确的"；基本不可用 |

### 自动化检查维度

系统中Checker Skill的一致性检查包括以下自动化规则：

```
规则1: 人物关系对称性检查
  若 A.relations 中包含 B（关系为"兄妹"），则 B.relations 中必须包含 A（关系也为"兄妹"）

规则2: 时间线顺序检查
  对时间线中所有标记了时间戳的事件，验证它们的时间顺序与事件间的因果关系一致

规则3: 能力-等级一致性检查
  若角色的 skill_level 为 "初学者" 或 "入门"，则其 skills 列表中不应出现 "大师" 级别的技能

规则4: 组织归属唯一性检查
  同一角色在不同卡中的 faction/organization 字段应一致

规则5: 死/生状态一致性检查
  若某角色被标记为 deceased: true 且 death_event_id 指向某事件，则时间线中该事件之后不应出现该角色的主动行为
```

### 项目数据

- **一致性检查命中率：约75%**（Checker发现的一致性问题中，75%确为真实问题，25%为误报——通常是Checker将合理的"角色成长变化"误判为矛盾）

## 维度4：创意丰富度 (Creativity)

**权重：15%**

### 定义

衡量设定包在canon基础上提供的合理延伸和创新。注意：创意丰富度**不是**鼓励脱离canon的随意发挥，而是评估：

- 在canon留白处的合理填充
- 对潜在线索的挖掘和展开
- 多可能性的探索（Storm模式的特色能力）
- 设定之间的有机联动和化学反应
- 为写作者提供"灵感触发点"的能力

### 评分 Rubric

| 分数 | 标准 | 典型描述 |
|------|------|----------|
| **5** | 深度创意，有机联动 | 在canon基础上提出多个有深度的延伸设定；设定之间形成有机联动（如人物关系变化引发组织格局变化）；Storm模式提供3+个各有特色的可能性分支；设定能激发写作者"想写"的冲动 |
| **4** | 有意义的延伸 | 在关键canon留白处提供了合理的创造性填充；至少2个设定元素之间有巧妙的关联设计；提供了有启发性但不跳脱的创意方向 |
| **3** | 有创意但不突出 | 存在一些创意点但深度不足；大部分延伸较为常规（模板化推测）；设定之间的联动少或浅 |
| **2** | 创意匮乏 | 大部分延伸是常见的套路化推测；设定之间几乎无联动；Storm模式未体现出与Clear模式的差异 |
| **1** | 无创意或胡乱创意 | 几乎没有canon之外的延伸（仅机械复述源材料）；或创意完全脱离canon，变成同人设定而非辅助资料 |

### 创意评估子维度

| 子维度 | 权重 | 说明 |
|--------|------|------|
| 留白填充质量 | 35% | 对canon中未明确但有需求的信息点的填充是否合理且有新意 |
| 线索挖掘深度 | 25% | 是否发现了源材料中的隐含线索并合理展开 |
| 联动设计 | 25% | 不同设定元素之间的关联是否巧妙、有机 |
| 灵感价值 | 15% | 设定是否具备激发写作者创作欲望的能力 |

## 维度5：格式规范性 (Format)

**权重：10%**

### 定义

衡量输出的结构化程度、可读性和工程规范性。这一维度虽然权重较低，但直接关系到系统的工程可靠性和用户体验。

### 评分 Rubric

| 分数 | 标准 | 典型描述 |
|------|------|----------|
| **5** | 结构完美，开箱即用 | 所有字段完整且类型正确；层次结构清晰（卡片->类别->条目）；命名规范统一；Markdown/JSON格式无瑕疵；可直接被下游工具链消费 |
| **4** | 结构良好，偶有瑕疵 | 核心结构完整；偶有非关键字段的格式问题（如一个非必填字段缺失）；不影响整体的可读性和工具链消费 |
| **3** | 基本规范但不精细 | 主要结构存在；字段命名基本统一；但存在多余空字段、不一致的缩进、部分嵌套层级过深等问题；需轻度人工整理 |
| **2** | 结构松散，影响使用 | 字段名不统一；嵌套混乱；部分关键字段缺失或类型错误；需要中度人工整理后才能使用 |
| **1** | 结构混乱 | 几乎没有统一结构；字段随意命名；无法被工具链消费；需要大量人工整理甚至重新组织 |

### 自动化检查项

```python
format_checks = {
    "schema_valid": "设定包整体是否符合顶层Schema定义",
    "field_completeness": "所有必填字段的填充率",
    "field_type_correctness": "字段值类型是否与Schema一致",
    "naming_convention": "字段命名是否遵循camelCase/snake_case统一约定",
    "card_id_uniqueness": "所有card_id是否全局唯一",
    "cross_reference_validity": "跨卡引用ID是否都能找到目标",
    "markdown_lint": "Markdown格式是否符合规范（若有Markdown输出）",
}
```

## 维度6：可用性/可复用性 (Usability)

**权重：10%**

### 定义

衡量设定包/卡是否便于写作者在后续创作中直接使用。核心关注：

- 设定信息是否按写作场景组织（而非按数据来源组织）
- 查找和引用是否便捷
- 能否快速定位到某个人物/事件/地点的完整信息
- 设定包是否支持增量更新（新增一张卡而不需要重新生成全部）
- 导出格式是否兼容常见写作工具（如Obsidian、Notion、Scrivener）

### 评分 Rubric

| 分数 | 标准 | 典型描述 |
|------|------|----------|
| **5** | 即拿即用，高度可复用 | 按创作场景组织（如"写战斗场面需要的设定"）；有快速索引；支持单卡导出/导入；与至少2种主流写作工具格式兼容；支持增量更新 |
| **4** | 基本好用，少量不便 | 按维度组织清晰；有基本索引；支持部分导出；与至少1种主流工具兼容；部分支持增量更新 |
| **3** | 可用但需要适应 | 信息组织合理但偏向"数据库视角"而非"写作视角"；检索需要浏览多张卡；导出格式有限；增量更新需要手动处理 |
| **2** | 使用不便 | 信息组织不符合写作习惯；查找需要翻阅大量内容；无导出兼容性；修改一处需要重新生成整体 |
| **1** | 难以使用 | 信息散乱无组织；无法定位所需信息；无导出能力；不如直接用搜索引擎 |

### 可用性子维度

| 子维度 | 权重 | 说明 |
|--------|------|------|
| 信息可发现性 | 30% | 写作者能否快速找到所需设定信息 |
| 写作场景适配 | 30% | 设定组织方式是否匹配实际写作流程 |
| 工具链兼容性 | 20% | 是否能导入常用写作工具 |
| 增量可维护性 | 20% | 是否支持部分更新而不破坏整体 |

## A/B对照测试设计

### 测试目标

通过与ChatGPT（通用大模型）进行同题对照，量化本Agent在写作辅助场景下的专业优势。

### 同题设计

#### 测试场景池（10个典型写作场景）

| 编号 | 场景类型 | 题材 | 核心需求 |
|------|----------|------|----------|
| T01 | 同人创作-人物补全 | 仙侠 | 基于原作配角创作完整人物设定 |
| T02 | 同人创作-世界观扩展 | 科幻 | 基于原作的科技体系扩展新分支 |
| T03 | 同人创作-时间线重构 | 历史向 | 基于原作事件重构平行时间线 |
| T04 | 原创-世界观搭建 | 奇幻 | 从零搭建完整的魔法世界观 |
| T05 | 原创-人物群像 | 都市 | 创建5+个互有关联的原创人物设定 |
| T06 | 原创-文化设定 | 架空历史 | 设计完整的架空文化体系 |
| T07 | 混合-设定补完 | 仙侠+科幻 | 为已有框架补充地理与科技设定 |
| T08 | 混合-冲突解决 | 奇幻 | 检测已有设定中的矛盾并提供修复方案 |
| T09 | 快速原型 | 不限 | 在15分钟内产出可用的最小设定包 |
| T10 | 深度研究 | 历史向 | 基于真实历史时期的创作背景研究 |

#### 输入标准化

每个测试场景提供标准化的输入包：

```json
{
  "scenario_id": "T01",
  "mode": "storm",  // storm | clear
  "canon_source": "《XXXX》原作第X章至第X章",
  "target_character": "原作中的配角XXX",
  "user_requirements": "需要完整的个人背景、性格分析、人际关系、潜在故事线",
  "output_format": "设定卡 + 设定包",
  "time_budget": "无限制"
}
```

### 评价方法：双通道评审

```
通道1: 人工评审（权重60%）
  ├── 评审团：3位资深写作者 + 2位AI产品评估专家
  ├── 每人独立对10个场景的A/B输出进行评分
  └── 评审意见录入标准化评分表

通道2: LLM评审（权重40%）
  ├── 使用独立的LLM（非本系统的底层模型）作为评审员
  ├── 每个场景进行3次独立评审（温度参数随机化以减少偏差）
  └── 取3次评审的均值
```

### 盲评设计

1. **输出匿名化**：本Agent和ChatGPT的输出经过统一的格式清洗，去除任何可识别来源的标记（如系统的emoji、特定的问候语、API特征等）。

2. **随机编号**：每份输出被随机分配一个代号（如`输出-A`、`输出-B`），评审员不知道代号与系统的对应关系。

3. **顺序随机化**：同一场景的两个输出（A/B）的展示顺序随机化，消除顺序偏差。

4. **揭盲流程**：所有评审完成后，独立第三方（非评审员）进行揭盲和数据汇总。

### 统计方法：多题多评委交叉验证

```
数据结构：10个场景 × 2个系统 × 5位人工评审 + 3次LLM评审

统计分析：
1. 描述统计：每个维度/系统的均值、中位数、标准差
2. 配对t检验：本Agent vs ChatGPT在各维度上的显著性差异
3. 评分者信度：Kendall's W 系数评估5位人工评审员的一致性
4. 人机一致性：人工评审均值 vs LLM评审均值的Pearson相关系数
```

### A/B对照结果速览（项目数据）

|------|---------|---------|------|

# 评估脚本架构（概念级）
class EvaluationPipeline:
    def __init__(self, test_scenarios: list[TestScenario]):
        self.scenarios = test_scenarios

    def run(self) -> EvaluationReport:
        results = []
        for scenario in self.scenarios:
            # Step 1: 生成输出
            agent_output = AgentSystem.generate(scenario)
            chatgpt_output = ChatGPTAPI.generate(scenario)

            # Step 2: 自动化评分
            auto_scores = {
                "coverage": self.auto_score_coverage(agent_output),
                "format": self.auto_score_format(agent_output),
                "consistency": self.auto_score_consistency(agent_output),
            }

            # Step 3: LLM评审
            llm_scores = self.llm_judge(
                output_a=agent_output,
                output_b=chatgpt_output,
                dimensions=["accuracy", "creativity", "usability"],
                num_trials=3
            )

            # Step 4: 收集结果
            results.append(ScenarioResult(
                scenario_id=scenario.id,
                auto_scores=auto_scores,
                llm_scores=llm_scores,
                human_scores=None  # 待人工评审填充
            ))

        return EvaluationReport(results)

    def auto_score_coverage(self, output: SettingPackage) -> float:
        """自动统计维度覆盖率"""
        dimensions = ["character", "worldview", "timeline", "geography",
                       "culture", "technology", "organization", "item"]
        covered = sum(1 for d in dimensions
                      if output.get_entries_count(d) >= 3)
        return (covered / len(dimensions)) * 5  # 映射到1-5分

    def auto_score_format(self, output: SettingPackage) -> float:
        """自动检查格式规范"""
        checks = {
            "schema_valid": output.validate_schema(),
            "required_fields_complete": output.check_required_fields(),
            "field_types_correct": output.check_field_types(),
            "card_ids_unique": output.check_card_id_uniqueness(),
            "cross_refs_valid": output.check_cross_references(),
        }
        passed = sum(1 for v in checks.values() if v)
        return (passed / len(checks)) * 5  # 映射到1-5分
